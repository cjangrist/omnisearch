# AGENTS.md â€” src/server/

Tool and REST orchestration. Three orchestrators (search fanout, AI answer fanout, fetch waterfall), three REST endpoints, and the MCP tool registry.

## Files

- **`tools.ts`** â€” MCP tool registration. Defines a `ToolRegistry` singleton (module-scoped) holding the three unified providers. `register_tools(server, get_ctx)` wires three tools â€” `web_search`, `answer`, `fetch` â€” onto an `McpServer`. Each tool callback closes over the per-DO `get_ctx` getter (R4F01) so trace `flush_background` writes attach to the DO's `waitUntil`. Tool input/output schemas use Zod. Long descriptions in the tool annotation tell agents when to prefer `web_search` over `answer` and how to use `skip_providers` on `fetch`. The `web_search` tool also accepts a `grounded_snippets?: boolean` input flag and returns each `web_results[i].snippet_source` âˆˆ `{ aggregated, grounded, fallback }` so callers can see which path produced each snippet.
- **`handlers.ts`** â€” MCP resource handlers (provider-status, provider-info). Read-only resources reflecting the active set.
- **`web_search_fanout.ts`** â€” `run_web_search_fanout(unified, query, options)` parallel-fans-out the query to every active search provider. Calls `rank_and_merge` (RRF + dedup + tail rescue + quality filter) and `collapse_snippets`. After ranking, when `GROQ_API_KEY` is set and a `fetch_provider` has been threaded through (via `tools.ts` / `rest_search.ts`), the top-N (currently `GROUNDING_TOP_N = 15`) URLs are passed to `ground_top_results` from `grounded_snippets.ts`; grounded snippets replace the engine-supplied ones and the rest of the ranked list is appended unchanged. Per-call opt-out via `options.grounded_snippets === false`. KV cache key is `hash_key('search:', query [+ '\0sqf=true'] [+ '\0gnd=true'])` â€” the grounded suffix prevents cross-mode collisions so a `grounded:false` raw call can't poison subsequent grounded calls or vice versa. Optional `timeout_ms` aborts pending providers at the deadline. Only complete fanouts (no failures, no aborts) are cached.
- **`grounded_snippets.ts`** â€” orchestrates the grounding stage. Public entry: `ground_top_results(query, results, fetch_provider, signal?)`. Bounded worker pool (default `concurrency = 3`) â€” each worker fetches a URL through `run_fetch_race` (the existing waterfall) and calls Groq (`openai/gpt-oss-120b`) via `http_json` to write a query-framed snippet. Per-URL hard deadline (default `per_url_deadline_ms = 15000`) fires an `AbortController` whose signal is threaded into the inner Groq call so a slow URL never stalls a worker past the cap. Pre-Groq junk-content detection (paywall / login-wall / cookie-wall / JS-shell / bot-challenge â€” see `grounded_prompts.ts`) triggers a single retry through the waterfall with `skip_providers={attempt-1 winner}`; sentinel detection in the model output is unified with the junk path for retry purposes. No retry on Groq verdict alone, no retry when `run_fetch_race` throws (waterfall already exhausted). Outcomes are classified into 8 buckets (`GroundingOutcome`: `grounded`, `grounded_via_retry`, `fallback:fetch_exhausted`, `fallback:fetch_too_short`, `fallback:fetch_junk_after_retry`, `fallback:groq_sentinel`, `fallback:groq_error`, `fallback:groq_empty`, `fallback:pipeline_timeout`). Emits structured per-phase logs (`op: 'grounding_phase'`) plus a single `grounding_aggregate` line per call with grounded ratio, p50/p95/max latency, provider wins, retry count, and timeout count. Also runs `close_unbalanced_fence` over the snippet so a model truncation mid-fence can't bleed downstream markdown. Successful snippets set `result.snippet_source = 'grounded'`; failures set `'fallback'` and leave the original aggregated snippet in place.
- **`grounded_prompts.ts`** â€” prompt + heuristic surface kept separate from the orchestrator so prompt iteration is low-blast-radius. Exports `GROUNDED_SYSTEM_PROMPT` (content-type-routed snippet writer with INVARIANTS â€” anti-preface forbidden-opener list, prompt-injection defense, language preservation), `build_grounded_user_message(query, title, content, max_chars)`, `GROUNDED_SNIPPET_MAX_CHARS = 2000`, and the junk + sentinel detectors. Junk patterns are split into `GROUNDED_JUNK_TIGHT_PATTERNS` (always fire) and `GROUNDED_JUNK_AMBIGUOUS_PATTERNS` (length-gated to â‰¤ 3000 chars so long-form prose mentioning "access denied" or "security check" rhetorically isn't retried for nothing). Sentinels (`[no usable content]`, `[navigation only]`, `[page not found]`, `[search results page]`, `[login required]`) â€” `detect_grounded_sentinel` strips trailing punctuation + markdown wrap chars before exact match, then falls back to a length-gated (â‰¤ 200 chars) substring match for prose-framed sentinels.
- **`answer_orchestrator.ts`** â€” `run_answer_fanout(ai_unified, web_unified, query)` builds one task per active AI provider. If `GEMINI_GROUNDED_API_KEY` is set AND a web search provider exists, an extra `gemini-grounded` task is added: it runs an inline 10-second `web_search_fanout`, then calls `gemini_grounded_search(query, sources, signal)`. Hard global deadline `GLOBAL_TIMEOUT_MS = 295_000` (4m55s); pending providers are aborted via `AbortController`. Logs progress every 5s. Only complete fanouts (zero failed providers) are cached.
- **`fetch_orchestrator.ts`** â€” `run_fetch_race(unified, url, options)` walks the fetch waterfall. The `CONFIG` literal at the top defines:
  - `breakers`: `github` (github.com, gist.github.com, raw.githubusercontent.com), `youtube` â†’ `supadata`, `social_media` â†’ `sociavault`.
  - `waterfall`: `solo: tavily` â†’ `solo: firecrawl` â†’ `solo: kimi` â†’ `parallel: [linkup, cloudflare_browser]` â†’ `parallel: [diffbot, olostep]` â†’ `parallel: [scrapfly, scrapedo, decodo]` â†’ `solo: zyte` â†’ `solo: brightdata` â†’ `sequential: [jina, spider, you, scrapeless, scrapingbee, scrapegraphai, scrappey, scrapingant, oxylabs, scraperapi, leadmagic, opengraph]`.
  - `failure`: `min_content_chars: 200` and `challenge_patterns` (cf-browser-verification, captcha, just a moment, etc.). API-native providers (`github`, `supadata`) are exempt.
  - Also exports `parse_skip_providers` (string | string[] â†’ string[]) and `validate_skip_providers` (returns `{ unknown }` to be 400'd at the entry layer).
- **`rest_search.ts`** â€” `POST /search`. Body: `{ query, count?, raw?, grounded_snippets? }`. `count` clamped to 0..100 (0 = all). `raw: true` skips quality filtering. `grounded_snippets: false` opts the call out of the Groq grounding stage (default ON when `GROQ_API_KEY` is configured). Returns `[{ link, title, snippet }]`. 502 + `{ error, failed_providers }` if all providers failed.
- **`rest_fetch.ts`** â€” `POST /fetch`. Body: `{ url, provider?, skip_cache?, skip_providers? }`. `provider` and `skip_providers` are mutually exclusive (400 if both). `provider` forces a single fetch provider, bypassing the waterfall. `skip_cache` bypasses cache reads only. `skip_providers` bypasses both cache reads and writes, and triggers a 2-provider compare returning the second under `alternative_results`.
- **`rest_researcher.ts`** â€” `GET` or `POST /researcher`. Auth via `Authorization: Bearer ...` OR `?api_key=...` query param. Body or query param `query`. Returns up to 10 search snippets shaped as `[{ href, body }]` for GPT-Researcher's `RETRIEVER=custom` integration. **Snippets only â€” no full-page fetch happens** (the file's older header comment is misleading; the implementation calls only `run_web_search_fanout`).

## Conventions / Invariants

- **All three orchestrators wrap their work in `run_with_trace(ctx, fn)`** so providers can call `get_active_trace()?.record_*` without threading.
- **Failure isolation per provider** â€” every fanout / waterfall step has its own try/catch. One provider's exception never crashes the rest.
- **Partial fanouts are NOT cached** â€” the cache validator stores `query` / `requested_url`, but partial fanouts skip the write entirely.
- **Cancellation propagates**: `make_signal(timeout, external)` combines an external signal with the per-provider timeout. The fanout's deadline is one such external signal.
- **`is_done` flag pattern** for late arrivals â€” once the orchestrator settles, late-arriving promises check `is_done` and skip mutating result arrays. Same pattern in `fetch_orchestrator.run_parallel` via the `resolved` flag, also dropping the loser's failure entry from `ctx.failed` AND the trace so the public response and trace tell the same story.
- **Error model**: throw `ProviderError` from inside provider implementations; orchestrators convert to `providers_failed` entries. REST handlers convert top-level `ProviderError(INVALID_INPUT)` to 400 and others to 502.

## Gotchas / History

- **`config.yaml` is documentation only.** The runtime fetch waterfall is the `CONFIG` literal at the top of `fetch_orchestrator.ts`. The two have drifted (YAML lacks the github breaker and the kimi waterfall step). Edit the TS first, then regenerate the YAML.
- **The `kimi` waterfall step requires both `KIMI_API_KEY` AND `SCRAPFLY_API_KEY`.** If only one is set, kimi is silently skipped at the active-set filter â€” correct behavior, but the dependency isn't reflected in `config.yaml`.
- **`gemini-grounded` is NOT in the unified AI registry.** `answer_orchestrator.ts` imports `gemini_grounded_search` directly and adds it as an extra task when web_search is also available. Its trace links to the inner `web_search` trace via `parent_trace_id`. Setting `GEMINI_GROUNDED_API_KEY` alone is enough; it does not need a registration line.
- **`/researcher` returns snippets, not full content** despite an older header comment. If you need full extraction, call `/fetch` separately or the MCP `fetch` tool.
- **Multi-winner parallel race**: when `skip_providers` triggers `target_count = 2`, the parallel step settles when 2 winners arrive (not on `Promise.any`). Late losers' rejections are dropped from both `ctx.failed` and the trace.
- **Empty-active-set guard on fetch**: if every active fetch provider is filtered out by `skip_providers`, `run_fetch_race` throws `INVALID_INPUT` (REST â†’ 400) rather than running the waterfall to exhaustion and emitting a misleading 502 with `"Tried: <empty>"`.
- **R4F01 (per-DO ctx capture)** is a tools.ts-specific invariant. The registry is module-scoped (shared across DO instances in the same isolate); each tool closure captures its own `get_ctx` so trace flush_background calls scope to the right DO. See README's "Concurrency model" section.
- **All-failed â†’ 502, not 200** on REST `/search` and `/researcher`. (R3F03)
- **Grounded snippets default to ON when `GROQ_API_KEY` is set.** Both the MCP `web_search` tool and `POST /search` accept a `grounded_snippets: false` opt-out. The fetch_provider needed by the grounding stage is threaded through `tools.ts` / `rest_search.ts` from the registry â€” `web_search_fanout` itself never reaches into the registry, so the orchestrator stays callable from contexts (e.g. answer fanout's gemini-grounded inline call) where grounding shouldn't run. The cache key is salted with `\0gnd=true` so grounded and non-grounded modes share neither cache hits nor poisoning paths.
- **`config.yaml` does not describe the grounding waterfall** â€” grounding uses Groq directly via `http_json`, not the fetch-orchestrator waterfall. The fetch waterfall *is* used inside each grounding worker (to pull each URL's content), but the grounding step itself is a separate orchestration in `grounded_snippets.ts`.
